# 1. `docs/itir_model.md` (FULL)

```md
# ITIR Data Model

## Purpose

ITIR (Investigative & Interpretive Reasoning) defines the **explicit interpretation layer** over the SensibLaw (SL) structural substrate.

ITIR objects represent **claims, hypotheses, actors, events, and relationships** that are:
- non-authoritative
- explicitly attributed
- reversible
- span-backed by SL

ITIR never mutates source text or structural spans.  
All interpretation remains an overlay.

---

## Core Principle

> **Every ITIR object must cite SL spans.  
> No interpretation exists without provenance.**

---

## Object Classes

### Claim

A Claim represents an asserted statement about the world.

Claims may be true, false, disputed, or unknown.

**Required fields**
- `claim_id`
- `asserted_by` (user, org, branch)
- `span_refs` (list of `(doc_id, start, end)`)
- `confidence` (0–1 or enum)
- `created_at`

**Optional**
- `summary` (derived, non-authoritative)
- `notes`

Claims do not imply correctness.

---

### Hypothesis

A Hypothesis is a tentative explanatory model.

Types include:
- `actor`
- `event`
- `motive`
- `timeline`
- `relationship`

**Required**
- `hypothesis_id`
- `type`
- `description`
- `supporting_claims`
- `contradicting_claims`
- `confidence`

Hypotheses may coexist and conflict.

---

### Actor (Hypothesis subtype)

Actors represent **identity hypotheses**, not resolved entities.

**Fields**
- `actor_id`
- `labels` (names, aliases, spellings)
- `evidence_spans`
- `confidence`

Actors may be merged or split later; history is preserved.

---

### Event (Hypothesis subtype)

Represents a hypothesised occurrence.

**Fields**
- `event_id`
- `description`
- `time_bounds` (optional)
- `evidence_spans`
- `confidence`

---

### Relationship

Directed relationships between ITIR objects.

**Types**
- `supports`
- `contradicts`
- `implies`
- `associated_with`
- `uncertain`

**Required**
- `source_id`
- `target_id`
- `relation_type`
- `provenance_spans`
- `confidence`

---

## Redactions & Uncertainty

Redactions are **structural spans in SL**.

In ITIR:
- redactions may anchor hypotheses
- uncertainty must be explicit
- no filling or guessing allowed

Example:
> “Actor A appears adjacent to repeated redactions in documents X, Y.”

---

## Branching & Attribution

All ITIR objects belong to a **branch**.

Branches:
- may diverge
- may be merged
- never overwrite history

Each object records:
- creator
- branch
- timestamp

---

## Invariants

- No ITIR object without span provenance
- No silent merges
- No inferred truth
- No mutation of SL text
- Deterministic references

---

## Non-goals

ITIR does NOT:
- decide truth
- rank narratives
- resolve guilt
- enforce consensus
- overwrite sources

---

## Summary

ITIR enables **explicit, auditable disagreement** over a shared factual substrate.

SL guarantees everyone sees the same text.  
ITIR allows them to argue responsibly.
```

Here’s the current situation on **programmatic access to AustLII/JADE**—specifically whether they require **API keys** or are **paid**:

---

### 🔹 AustLII (Australasian Legal Information Institute)

**1. Public & Free Access**  
AustLII’s core legal data (cases, legislation, treaties, law reform materials etc.) is freely available on the web for users. AustLII’s mission is to provide **free public access to law**, not a paid‐access service. citeturn1search3

**2. Programmatic Access & APIs**  
AustLII doesn’t offer an official, modern REST/JSON API with API keys like many commercial services. Instead:

- AustLII publishes documentation for **developers** to interface with its search and content systems via the **SINO CGI API**. This is essentially a CGI search interface using HTTP parameters and forms that your application can call. citeturn1search0turn1search4  
- The SINO CGI API can be used to build search forms and programmatically query the search engine. It’s a **public, documented interface** but it’s essentially HTTP form/query based—not a key/token API. citeturn1search0  
- There’s **no official requirement for an API key**, paid subscription, or commercial account to use this search API form. citeturn1search0

**3. Practical Constraints**  
Because the interface is built on older CGI scripts:

- You’ll typically be querying via URLs like `https://www.austlii.edu.au/cgi-bin/sinosrch.cgi?...` with search parameters.  
- This can be done programmatically (e.g., via script) without a key.  
- However, you should respect AustLII’s **usage policies and robots.txt**, and reasonable request rates, because excessive automated scraping may be blocked or violate terms.

> **Summary:** AustLII *does* support programmatic access through its SINO CGI interface, and **no API key or paid plan is required** to use it. It’s not a modern API with tokens but rather a **HTTP query interface** that developers can embed or script against. citeturn1search0

---

### 🔹 JADE (BarNet JADE / JADE Legal Research Platform)

JADE is a separate legal research platform from AustLII (often with richer features, alerts, and blogging tools). Based on available documentation:

- JADE primarily supports users via an interactive legal research interface; in older documentation you can construct URLs for content. citeturn0search1  
- There isn’t a clearly published **official programmatic API** for JADE like a JSON API with keys in the public documentation. citeturn0search1  
- Some services around JADE (e.g., **JADE Pro alerts**) might be part of a *premium offering* in other products (like OpenLaw’s integrations), but AustLII itself doesn’t govern that. citeturn0search13

> **In short:** For JADE, there’s **no standard documented API key / paid API** from AustLII/AustLII-linked sources available publicly—if there’s a paid or key-protected API it’s likely provided by JADE’s own service operators (e.g., BarNet or partner products), not AustLII.

---

### 🧠 Best Practice for Programmatic Use

If you want stable, robust programmatic access to AustLII content:

Great question — this is the right moment to pin it down *mathematically*, otherwise “GPU for law” sounds like cargo culting.

I’ll answer in two layers:

1. **What these text/string methods *are*, mathematically**
2. **What you actually gain by running parts of them through dashiCORE / GPU**

No fluff, no hype.

---

## 1. What these are, mathematically

### 1️⃣ Text is **not language** here — it is a **finite relational structure**

Once you strip away NLP instincts, everything you’re proposing lives in a very classical space:

> **Finite labelled graphs + automata + constraints**

Concretely:

### (a) Strings → symbols → spans

Let  
- Σ be a finite alphabet (Unicode after normalization)
- A document is a string \( s \in \Sigma^\* \)

After normalization, you do **not** work on \( s \) directly.

You map it to a **finite set of spans**:
\[
\mathcal{S} = \{(i, j, \ell)\}
\]
where:
- \( i,j \) are character offsets
- \( \ell \) is a label (TOKEN, SECTION, NUMBER, CASE\_CITATION, etc.)

This is a **regular-language operation** (regex + DFA/PDA).

So far:  
📐 *formal language theory, nothing fancy*

---

### (b) Spans induce a **directed, labelled graph**

Define a graph:
\[
G = (V, E)
\]

- \( V = \mathcal{S} \)
- \( E \subseteq V \times L \times V \)

Edge labels include:
- adjacency
- containment
- reference
- modification
- exception

This graph is:
- finite
- deterministic
- fully observable

📐 *this is a labelled relational structure*

---

### (c) Logic trees / obligation graphs are **quotients of this graph**

Your logic tree construction is mathematically a **quotient**:
\[
\pi : G \rightarrow G / \sim
\]

Where \( \sim \) identifies spans that form a single legal operator:
- obligation
- exception
- definition
- condition

This is exactly:
- graph contraction
- subtree extraction
- partial order construction

📐 *category theory / graph quotients / term graphs*

---

### (d) Admissibility = **automata + constraints**

Your admissibility rules (“this clause is valid / ambiguous / invalid”) are:

- A **pushdown automaton** over the span graph
- Or equivalently, a **constraint satisfaction problem**

Each node gets a state:
\[
x_v \in \{-1, 0, +1\}
\]

- −1 reject
- 0 project / ambiguous
- +1 accept

Transitions are local and structural.

📐 *this is a signed automaton on a finite graph*

---

### (e) MDL / compression = **minimum description length on graphs**

When you:
- canonicalize obligations
- hash obligation signatures
- merge equivalent clauses

You are minimizing:
\[
L(\text{model}) + L(\text{data} \mid \text{model})
\]

That is **exactly** MDL — but over **symbolic graph structure**, not pixels.

📐 *information theory on relational objects*

---

### Summary of the math

What you have is:

> **A finite labelled graph + a signed automaton + an MDL objective**

This is *the same mathematical class* as:

- constraint graphs
- SAT/SMT cores
- compiler IRs
- database query plans
- static analyzers

Not language models. Not NLP.

---

## 2. What do we gain from GPU / dashiCORE?

Now the important part: **why bring dashiCORE / GPU into this at all?**

### Short answer

> You gain **scale, determinism, and global relational visibility** — *not* “semantic intelligence”.

Let’s be precise.

---

## 2.1 What dashiCORE gives you abstractly

dashiCORE is good at exactly one thing:

> **Fast, deterministic operations on large relational state spaces**

Formally:
- adjacency matrices
- incidence tensors
- signed fields
- constraint propagation
- repeated local updates until convergence

That maps *perfectly* onto:

- obligation graphs
- cross-document references
- jurisdiction-wide rule sets
- diffing thousands of statutes at once

---

## 2.2 Why CPU-only breaks down

CPU symbolic systems struggle when:

- relations are dense
- comparisons are pairwise
- updates are iterative
- global fixed points matter

Example:
- “Which obligations across 40 Acts conflict under these conditions?”
- “Which amendments invalidate which downstream duties?”
- “Which clauses survive admissibility projection?”

Those are:
\[
O(n^2) \text{ to } O(n^3)
\]
graph problems.

You *can* do them on CPU — but not interactively, not at national scale.

---

## 2.3 What GPU changes (the real gain)

### (a) Massive parallel constraint propagation

GPU excels at:
- apply the *same local rule* everywhere
- over a large graph
- repeatedly

That’s literally:
\[
x^{(t+1)} = \mathcal{P}(A x^{(t)})
\]

Where:
- \( A \) = adjacency / relation tensor
- \( \mathcal{P} \) = admissibility projector (your PDA rule)

This is the **same algebraic shape** as your CFD proxy update.

---

### (b) Deterministic global fixed points

Instead of:
- local heuristics
- order-dependent evaluation

You get:
- whole-system convergence
- reproducible states
- hashable results

This matters legally:
- auditability
- appeals
- compliance tooling
- regression testing across law versions

---

### (c) Structural similarity at scale

Comparing two obligations is cheap.

Comparing:
- 10,000 obligations × 10,000 obligations  
is **not** cheap — unless you batch it.

GPU lets you do:
- adjacency tensor comparisons
- signature matrix multiplications
- diff masks

All without “understanding” language.

---

### (d) Jurisdiction-wide MDL

MDL selection over:
- millions of clauses
- thousands of amendments

is a **global optimization problem**, not local reasoning.

GPU lets you:
- score many compressions in parallel
- find stable canonical forms
- detect redundancy *system-wide*

---

## 2.4 What GPU does *not* give you (important)

GPU does **not** give you:
- interpretation
- intent
- fairness
- meaning
- policy judgment

Those remain:
- human
- procedural
- institutional

GPU gives you:
> **fast, correct, explainable structure**

Nothing more, nothing less.

---

## 3. The unifying statement (this is the key insight)

Here’s the clean way to say it:

> **SensibLaw is a symbolic system whose state space is a graph.

- admissibility projection

That’s why this isn’t a stretch — it’s a reuse.
