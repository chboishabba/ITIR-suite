# Moonshot Compiler Normalization Reconsideration

Date: 2026-04-02

## Purpose

Reconsider the earlier judgment-architecture split in light of the broader
ITIR moonshot.

The previous pass was directionally right, but it was too Nat-centric. It
framed the next work mainly as a Wikidata moonshot support program plus
adjacent legal/text/graph inputs.

That is incomplete.

For the broader ITIR moonshot, AU and GWB should be read through the same
normalization lens:

- bounded evidence group in
- deterministic extraction and promotion
- promoted review/product outcomes out
- graph as a derived inspection surface, not the main product

## Read

The receipts-first compiler spine already says the stronger cross-lane shape:

1. source substrate
2. deterministic extraction
3. promotion boundary
4. reasoning and graph
5. public-action packaging

That means the reusable target is not "do Nat helpers, then maybe legal."
The reusable target is:

- define what evidence bundle a lane accepts
- define what promoted facts or abstentions it emits
- define what review/product artifacts it emits from that promoted state
- keep graph and narrative layers derived and challengeable

In plain terms:

- AU should move toward: hearing/legal evidence bundle in -> promoted
  procedural/representation/review outcomes out
- GWB should move toward: public-source evidence bundle in -> promoted
  action/relation/review outcomes out
- Wikidata should move toward: revision/text/evidence bundle in -> promoted
  migration/split/hold outcomes out

## Outcome Of The Previous Pass

The previous moonshot revision pass still produced useful decisions:

- stronger judgment layers must stay grounded, auditable, and fail-closed
- text does not bypass promotion
- graph must not silently become truth
- typed primitives and explicit abstain/conflict handling remain important

What it missed was the product/compiler framing across AU and GWB.

It overweighted:

- Nat helper depth
- claim-boundary support
- graph discipline as a separate lane

It underweighted:

- shared evidence-group -> promoted-outcome contracts
- reusable product surfaces across AU, GWB, and Wikidata
- the fact that graph should usually be one of the emitted products, not the
  organizing abstraction

## Revised Main Decision

Reframe the next moonshot-adjacent round around compiler-shaped product
normalization.

The strongest reusable question is now:

- if you put bounded evidence group `X` into lane `L`, what promoted product,
  outcome, packet, graph, or action surface `Y` do you get back?

That should be explicit for:

- AU
- GWB
- Wikidata/Nat

And the reusable contract should separate:

- input evidence bundle contract
- promoted truth and abstention contract
- derived review/product surface contract

## Revised Priority Order

1. common evidence-bundle -> promoted-outcome contract
2. AU product normalization on that contract
3. GWB product normalization on that contract
4. Nat/Wikidata text bridge and hard-case grounding as one adopter family
5. tiny shared primitive/comparison layer, but only where it directly supports
   the product contracts
6. graph/claim-boundary overlays and governance gates as hardening layers

## Governance Read

- ITIL:
  - normalize the service contract first:
    what input a lane accepts and what output it reliably returns
- ISO 9000:
  - make output quality criteria and handoff contracts explicit
- ISO 42001 / NIST AI RMF:
  - keep grounded evidence, promoted truth, and derived outputs separate
- ISO 27001 / ISO 27701:
  - keep personal/protected evidence boundaries explicit when normalizing AU
    and similar lanes
- ISO 23894:
  - treat hidden promotion, graph overclaim, and output ambiguity as core
    risks
- Six Sigma:
  - reduce lane variance by making AU/GWB/Wikidata produce comparable classes
    of outputs from comparable classes of inputs
- C4:
  - containers should expose stable lane products; graph and reporting stay
    downstream of the promotion core
