# Suite Bounded Search And Uncertainty Collapse

Date: 2026-04-03

## Purpose

Record the next moonshot control read after the normalized-adapter phase.

The suite now has enough normalized substrate that the next priority is no
longer adapter proliferation for its own sake.

The next priority is using existing proving-ground lanes and large corpora to
determine:

- where the current graph or world model is weakest
- what bounded knowledge can be extracted from already-owned sources
- whether the suite can determine the next necessary search bounds
- which search or follow step should collapse uncertainty fastest with the
  highest likely authority yield

In short:

- inspect uncertainty
- decide whether local context already closes it
- if not, derive a bounded search or follow obligation
- prioritize the next search or follow step
- interpret the result back into promoted or derived products
- continue until all determinable facts in scope are exhausted

## What This Adds Above The Current Stack

The current normalized concepts remain valid.

This note adds one new control read above them:

- the suite should become able to determine necessary bounded search to fill
  graph or world-model gaps

That does not mean:

- search everything
- crawl without bounds
- ignore local context
- treat every statement as internet-search-triggering

It means:

1. inspect the current bounded product, graph, or state surface
2. identify the weakest or most consequential unresolved area
3. determine whether local sources already suffice
4. if not, derive a bounded follow or search obligation with:
   - trigger
   - scope
   - stop
   - likely authority target
   - expected uncertainty-collapse value
5. prioritize the next action by expected uncertainty collapse per unit cost
6. reintegrate results through the same promotion and derived-product rules

## Example Read

Some statements do not justify external search:

- personal or local statements where the suite already has the relevant
  context
- bounded internal legal interpretation where the cited material is already in
  hand

Some statements do justify bounded search:

- public known persons where identity or role grounding is missing
- legal references where cited authority exists but is not yet attached
- public-source or event claims where the current evidence is visibly partial
- ontology or entity cases where a known Wikidata, Wikipedia, or official
  record is likely to collapse ambiguity quickly

The key rule is:

- external search is not the default
- bounded search is derived from visible uncertainty and expected authority
  gain

## Constraints, Invariants, Preconditions, Postconditions

This note does not replace the root constraints.

It adds one important derived rule:

- the suite should search only when the current bounded context cannot answer
  the question and a bounded search is likely to reduce uncertainty with a
  justified authority target

Additional invariants:

- search-bounds invariant:
  every external follow or search action must be triggered by visible
  unresolved pressure, not curiosity alone
- authority-yield invariant:
  higher-authority and lower-cost collapse paths should be preferred over
  broad low-yield search
- local-first invariant:
  the suite should exhaust useful local context before deriving wider search
- canonical-live-context invariant:
  ChatGPT live-context refresh should route through the canonical
  `robust-context-fetch` / `chat_context_resolver` DB-first contract, with
  raw `reverse-engineered-chatgpt` pulls treated as a subordinate fetch
  mechanism rather than a separate moonshot lane or truth surface

Additional preconditions:

- the current artifact, graph, or state surface exposes enough unresolved
  pressure to justify a search
- the search target class is identifiable enough to bound
- the likely value of the search exceeds its churn and ambiguity cost
- if the missing context is a ChatGPT thread, the canonical archive lookup and
  resolver-based live fallback path are the first-class precondition before any
  direct `re_gpt` troubleshooting is treated as product work

Additional postconditions:

- the search decision is inspectable
- the chosen bounds are explainable
- the resulting evidence is reintegrated through normal promotion and derived
  paths
- unresolved pressure either shrinks or is made more explicit

## Immediate Orchestrator Read

The next high-leverage worker lanes should no longer be framed primarily as:

- what other adapter can we normalize?

They should be framed as:

- which existing proving ground is weakest?
- what uncertainty can be collapsed locally?
- what bounded search or follow should run next?
- how do we rank the next searches by authority yield and uncertainty
  reduction?

For chat-context retrieval specifically:

- use `robust-context-fetch` first
- only drop to direct `reverse-engineered-chatgpt` repair when the resolver
  contract itself is blocked or missing a required capability

That is the next suite-level moonshot driver.
