State the repo's current disciplined reading of motif, cohomology, and
legal-tree language so exploratory structure does not drift into hidden

## Main Decision
Treat motif / meme / cohomology language as exploratory or explanatory framing
unless and until it cashes out into:

- source anchors
- reversible transforms
- candidate state
- promotion basis
- explicit proofs or receipts where required

The authoritative doctrine remains:

`source -> annotation/signal -> candidate -> promoted -> projection`

## Motifs
`motif` is not currently a repo-wide first-class canonical object.

The safe current reading is:

- motifs may appear as candidate-level or overlay-level structure
- motifs may help discover repeated patterns, compressions, or reusable
  semantic shapes
- motifs are non-authoritative until a lane defines explicit promotion rules

So the repo should read:

`motif != truth-bearing record`

and instead:

`motif = candidate or diagnostic structure below promotion`

## MotifCandidate Shape
If a lane wants motifs to participate in governed truth construction, the
minimum acceptable shape is a bounded `MotifCandidate`-like object carrying:

- source span or object anchors
- reversible derivation / transform history
- evidence or support refs
- conflict or overlap diagnostics
- promotion basis sufficient for review

Without that, motif language remains interpretation, not contract.

## Cohomology
The repo does not currently grant cohomology a formal truth-bearing role.

At most, it is admissible as:

- candidate-graph analysis
- overlap/conflict diagnostics
- clustering or invariance hints
- guidance for review or promotion pressure

It is not currently admissible as:

- a direct truth definition
- a substitute for anchors or promotion
- a way to bypass source-linked review

So the conservative reading is:

`candidate/overlay analysis -> maybe cohomology`

not:

`cohomology -> canonical truth`

## Legal Tree
The repo already uses legal-tree language for deterministic structured logic
over actors, obligations, conditions, references, and related elements.

But the legal tree is not automatically equivalent to promoted truth in every
lane. The safe interpretation is:

- legal-tree structures must remain source-anchored and reproducible
- some tree nodes may be annotation, candidate, or projection state
- only nodes or records explicitly backed by promotion are truth-bearing

So the repo should not assume:

`legal_tree subset promoted_truth`

as a blanket theorem.

Instead it should assume:

- legal-tree views may mix substrate-linked structure plus candidate or
  promoted material
- truth-bearing status depends on explicit promotion semantics for the
  relevant node family

## Safe Construction Sequence
If motif-style structure is introduced into legal-tree or reasoning work, the
safe sequence is:

1. derive candidate motifs or pattern structure from anchored source and
   reversible transforms
2. evaluate support, consistency, provenance, and review pressure
3. promote only what satisfies lane-specific gates
4. build higher reasoning or legal-tree claims only from promoted or
   explicitly typed non-canonical inputs

## Non-Goals
- not a commitment to add `MotifCandidate` schema immediately
- not a claim that cohomology is now part of the formal architecture
- not a claim that every legal-tree node is already promotion-backed

## Followthrough
- keep motif/cohomology discussion below promotion in roadmap and handoff docs
- if a future lane wants motif promotion, require:
  - anchor shape
  - transform provenance
  - review semantics
  - promotion gate definition
- if a future legal-tree schema is formalized, label which node families are:
  - substrate-linked only
  - candidate-only
  - promoted
  - projection-only
